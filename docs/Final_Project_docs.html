ğŸ“Š Project Overview

ğŸ¯ Mission: Build a production-ready multi-vendor e-commerce platform with ERP capabilities for sellers - think Amazon Seller Central + Flipkart Shopping Experience combined.

What is ShopMaster Pro?

ShopMaster Pro is a full-stack MERN application that enables multiple sellers to list their products on a single platform while customers can browse, purchase, and track orders. The admin controls seller approvals and platform operations.

ğŸ­ Three User Roles

ğŸ‘‘ Admin

CRITICAL

Platform owner who approves sellers, manages categories, and monitors analytics.

  • Approve/reject seller registrations
  • Create & manage product categories
  • View total revenue, orders, sellers

ğŸª Seller

CRITICAL

Vendor who sells products with full ERP dashboard for inventory & orders.

  • Manage product catalog (CRUD)
  • Track inventory (low stock alerts)
  • Fulfill orders & view revenue

ğŸ›ï¸ Customer

CRITICAL

End-user who browses products, adds to cart, and places orders.

  • Browse with filters (category, price)
  • Cart management
  • Checkout & order tracking

ğŸ¢ Real-World Analogy

Imagine a shopping mall:

  • Admin = Mall owner (decides which shops can open, manages mall rules)
  • Seller = Individual shop owner (manages their inventory, sales, revenue)
  • Customer = Shopper (visits different shops, buys products)

ğŸ” Authentication Flow

Registration Flow CRITICAL

Step 1: User submits registration form

Frontend sends: POST /api/auth/register

de>{
  "name": "John Doe",
  "email": "john@example.com",
  "password": "password123",
  "role": "customer"
}

Step 2: Backend generates OTP

Password is hashed with bcrypt (10 rounds). 6-digit OTP generated: Math.floor(100000 + Math.random() * 900000)

Step 3: OTP sent to email

Nodemailer sends email with OTP. OTP expires in 10 minutes (Date.now() + 10 * 60 * 1000)

Step 4: User enters OTP

Frontend sends: POST /api/auth/verify-otp

de>{
  "userId": "67abc123...",
  "otp": "123456"
}

Step 5: OTP verified, JWT issued

If OTP matches and not expired, user.isVerified = true. JWT token generated with payload:

de>jwt.sign(
  { id: user._id, role: user.role },
  process.env.JWT_SECRET,
  { expiresIn: '7d' }
)

Step 6: Token stored in frontend

Redux stores user data + token. LocalStorage stores both for persistence.

Login Flow CRITICAL

Step 1: User submits login form

Frontend sends: POST /api/auth/login

de>{
  "email": "john@example.com",
  "password": "password123"
}

Step 2: Backend verifies credentials

Check if user exists and isVerified = true. Compare password: bcrypt.compare(password, user.password)

Step 3: JWT token issued

Return token + user object (id, name, email, role)

Step 4: Frontend stores & redirects

Based on role:

  • admin â†’ /admin/dashboard
  • seller â†’ /seller/dashboard
  • customer â†’ / (homepage)

Protected Route Logic

de>// Every API request includes token

// Role-based access
if (requiredRole && req.user.role !== requiredRole) {
  return res.status(403).json({ msg: 'Access denied' });
}

Auth Slice CRITICAL

de>// src/redux/slices/authSlice.js
import { createSlice } from '@reduxjs/toolkit';

const authSlice = createSlice({
  name: 'auth',
  initialState: {
    user: JSON.parse(localStorage.getItem('user')) || null,
    token: localStorage.getItem('token') || null,
    isAuthenticated: !!localStorage.getItem('token')
  },
  reducers: {
    setUser: (state, action) => {
      state.user = action.payload.user;
      state.token = action.payload.token;
      state.isAuthenticated = true;
      localStorage.setItem('user', JSON.stringify(action.payload.user));
      localStorage.setItem('token', action.payload.token);
    },
    logout: (state) => {
      state.user = null;
      state.token = null;
      state.isAuthenticated = false;
      localStorage.removeItem('user');
      localStorage.removeItem('token');
    }
  }
});

export const { setUser, logout } = authSlice.actions;
export default authSlice.reducer;

Common Interview Questions & Answers

Q1: Why did you choose MERN stack?

Answer: "MERN stack mein JavaScript har layer mein use hoti hai - frontend se backend tak. MongoDB ka flexible schema multi-vendor project ke liye perfect hai kyunki different categories mein different attributes ho sakte hain. React ka component-based architecture code reusability deta hai. Node.js ka non-blocking I/O model high concurrent requests handle karne ke liye suitable hai, jo e-commerce mein zaroori hai."

Q2: How did you handle authentication?

Answer: "Do-layer authentication use kiya - pehle user register karta hai to email pe 6-digit OTP jaata hai (10-minute expiry). OTP verify hone ke baad JWT token issue hota hai jo 7 days valid rehta hai. Token mein user ID aur role encrypt karte hain. Har protected route pe middleware token verify karta hai aur role check karta hai. Security ke liye passwords bcrypt se hash karte hain (10 salt rounds)."

Q3: Explain the order flow in your system

Answer: "Customer products browse karta hai filters use karke. Cart mein add karta hai. Checkout pe address dalta hai. Order place karte waqt backend pehle stock check karta hai - agar sufficient hai to order create hota hai. Har item ke saath sellerId store hoti hai. Product stock atomically reduce hota hai. Cart clear ho jaata hai. Seller apne dashboard pe sirf apni products ke orders dekhta hai - items.sellerId filter se. Status update kar sakta hai: pending â†’ confirmed â†’ shipped â†’ delivered."

Q4: How does multi-vendor work?

Answer: "Order schema mein items array hai jisme har item ke saath sellerId reference store hota hai. Ek customer ka order multiple sellers ke products contain kar sakta hai. Jab seller apne orders fetch karta hai to backend filter lagata hai: items.sellerId === currentSeller. Isse har seller sirf apne products ke orders dekhta hai. Admin sabhi orders dekh sakta hai analytics ke liye."

Q5: What was the biggest challenge?

Answer: "Stock management sabse challenging tha. Race condition ho sakti thi - multiple users simultaneously same product order kar sakte the. Solve karne ke liye order placement mein synchronous stock check kiya. Pehle product fetch karte hain, stock verify karte hain, then atomically reduce karte hain with save(). Future mein MongoDB transactions use kar sakte hain for complete atomicity."

Answer: "Product schema flexible chahiye tha - Electronics category mein specs different hain (RAM, processor), Fashion mein size/color. NoSQL mein schema changes easy hain. Plus MongoDB Atlas ka free tier deployment ke liye convenient hai. Mongoose ODM se schema validation mil jaata hai. Agar strong transactions zaroori hote to PostgreSQL better choice hota."

Answer: "Redux Toolkit use kiya. Auth state globally chahiye tha - har component ko user details aur token access chahiye. Cart bhi multiple pages mein use hoti hai.
